# 디자인 QA 분석 도구 — 기획서

> Figma 디자인과 실제 구현된 웹이 일치하는지 한눈에 비교하는 디자인 QA 도구

작성일: 2026-06-13

---

## 1. 개요

| 항목 | 내용 |
|------|------|
| **목적** | Figma 디자인과 실제 웹 구현의 차이를 자동으로 찾아 디자인 QA를 효율화 |
| **한 줄 정의** | "Figma URL + 웹 URL → 차이 리포트" |
| **사용자** | 프로덕트 디자이너 (본인 + 공유 대상) |
| **핵심 원칙** | 가볍게 · 로그인 없음 · 클라우드 서버 없음 |

### 해결하려는 문제
디자인 QA를 지금은 **눈으로 일일이 대조**한다 — 색이 미묘하게 다른지, 패딩이 몇 px 틀어졌는지, 폰트 굵기가 맞는지. 느리고, 놓치기 쉽고, "느낌상 다른데 정확히 뭐가 다른지" 설명하기 어렵다. 이 도구는 그 대조를 자동화하고 **수치로** 보여준다.

---

## 2. 아키텍처 방향 (서버리스 · 로컬)

```
[내 컴퓨터에서 도는 로컬 도구]
        │
        ├─ Figma URL ──→ Figma REST API (개인 토큰) ──→ 프레임 이미지 + 노드 속성
        │
        ├─ 웹 URL ──────→ 로컬 헤드리스 브라우저(Playwright) ──→ 스크린샷 + 렌더링 스타일
        │
        └─ 비교 엔진 ──→ 자체 완결형 HTML 리포트 1장 생성
                                │
                                └─→ (선택) 정적 호스팅에 올려 "링크로 공유"
```

- **분석은 로컬**에서 (헤드리스 브라우저가 임의 웹페이지를 캡처해야 하므로 — 브라우저만으로는 CORS로 불가).
- **결과 리포트는 웹 링크로** 공유 가능 (자체 완결형 HTML → Netlify Drop / GitHub Pages / 파일 그대로).
- 클라우드 백엔드 0개. 이후 정말 "접속형 웹앱"이 필요해지면 **캡처 부분만** 서버리스로 분리하면 됨 (구조를 미리 그렇게 잡아둠).

---

## 3. 사용 흐름 (User Flow)

1. 로컬 도구를 한 번 실행해 둠 (`npm run dev` → 브라우저에 미니 UI가 뜸)
2. 입력칸에 **Figma 프레임 URL** + **웹 URL** 붙여넣기 → **[분석]** 클릭
3. 잠깐 캡처/비교 진행 (진행 표시)
4. **리포트** 화면 표시: 요약 점수 → 시각 오버레이 → 속성 차이 표
5. 필요하면 **리포트 HTML을 링크로 공유**

---

## 4. 기능 범위 (MVP)

비교는 두 축을 **모두** 제공. 단, 같은 캡처 데이터를 재사용하므로 무게는 거의 안 늘어남.

### A. 시각적 오버레이 diff
- Figma 캡처 vs 웹 캡처를 같은 폭으로 정규화 후 겹침
- **오버레이 슬라이더**(좌우로 끌어 비교) + **픽셀 차이 히트맵**
- 전체 일치율(%) 표시
- 라이브러리: `pixelmatch` (초경량)

### B. 속성 값 비교
- **느슨한 자동 매칭**: 위치·크기·텍스트가 비슷한 요소끼리 Figma↔웹 짝지음
- 비교 항목:
  - 색상 (배경/텍스트/테두리)
  - 타이포 (폰트 크기 / 굵기 / 행간 / 자간)
  - 간격 (패딩 / 마진 / 갭)
  - 크기 (너비 / 높이)
- 허용 오차(tolerance) 기준으로 **OK / 경고 / 불일치** 판정
- 매칭 애매한 요소는 **"확인 필요"** 로 표시하고 넘어감 (완벽 1:1 매칭은 과함)

---

## 5. 리포트 구성

자체 완결형 HTML 한 장. 위→아래 순서:

1. **요약 카드** — 종합 일치율, 불일치 N건 / 경고 N건, 분석 일시·대상 URL
2. **시각 비교** — Side-by-side + 오버레이 슬라이더 + 차이 히트맵
3. **속성 차이 표** — 심각도순 정렬
   | 요소 | 항목 | Figma | 웹 | 차이 | 판정 |
   |------|------|-------|-----|------|------|
   | 기본 버튼 | 패딩 | 16px | 12px | -4px | 🔴 불일치 |
   | 제목 | 폰트 굵기 | 700 | 600 | - | 🟡 경고 |
4. **확인 필요 목록** — 자동 매칭이 애매했던 요소들

---

## 6. 기술 스택 (가볍게)

| 영역 | 선택 | 이유 |
|------|------|------|
| 런타임 | Node.js | 로컬 실행 |
| 웹 캡처 | Playwright | 스크린샷 + computed style + 좌표를 한 번에 |
| Figma 데이터 | Figma REST API + 개인 액세스 토큰 | 이미지·속성 동시 취득, 독립 실행 가능 |
| 시각 diff | pixelmatch | 의존성 최소 |
| 미니 UI / 리포트 | 정적 HTML + 약간의 JS (또는 Vite) | 빌드 부담 최소 |

> Figma 토큰은 로컬 `.env`에 보관 (커밋 금지). 본인만 사용하므로 안전.

## 7. 알아둘 제약 / 엣지케이스

- **반응형**: 웹 캡처 시 뷰포트 폭을 Figma 프레임 폭에 맞춰야 정확함 → 입력에 "기준 폭" 옵션 추가
- **다크모드 / 상태**: hover·active 등 동적 상태는 기본 캡처 대상 아님 (추후 확장)
- **매칭 한계**: 자동 매칭은 완벽하지 않음 → "확인 필요"로 투명하게 표시
- **폰트 렌더링 차이**: OS/브라우저 렌더링 차이로 픽셀 diff에 미세 노이즈 → 허용 오차로 흡수
- **동적 콘텐츠/애니메이션**: 캡처 전 로딩 대기 + 애니메이션 정지 처리 필요

---

## 8. 개발 단계 (로드맵)

- **1단계 — 캡처 파이프라인**: Figma URL→이미지·속성, 웹 URL→스크린샷·스타일 추출
- **2단계 — 시각 diff**: 정규화 + pixelmatch + 오버레이 리포트
- **3단계 — 속성 비교**: 느슨한 매칭 + 항목별 비교 + 판정/허용오차
- **4단계 — 리포트 UI**: 요약·시각·표를 자체 완결형 HTML로
- **5단계 — 미니 입력 UI + 공유**: 입력칸 2개 + 분석 버튼, 리포트 링크 공유

---

## 9. 향후 확장 (지금은 안 함)

- 접속형 웹앱화 (캡처만 서버리스로 분리)
- 여러 프레임/페이지 일괄 분석
- 동적 상태(hover 등) 캡처
- 디자인 토큰 라이브러리 기준 검증
- 히스토리 비교 (이전 리포트 대비 변화)
